# Unit of Work 分割

## 依存関係図

```
                ┌─────────────────────────┐
                │  Unit 0: 共通基盤 + 認証  │
                │  （最初に完了させる）       │
                └───────────┬─────────────┘
                            │
              ┌─────────────┴─────────────┐
              │                           │
              ▼                           ▼
┌─────────────────────────┐  ┌─────────────────────────┐
│  Unit 1: 勤怠ドメイン     │  │  Unit 2: 社員管理        │
│  打刻・履歴・集計・修正   │  │  社員 CRUD（ADMIN）      │
│                         │  │                         │
│  担当: Person A         │  │  担当: Person B         │
└─────────────────────────┘  └─────────────────────────┘
        （並行実装可能 — 相互依存なし）
```

## 各 Unit サマリ

| Unit | 名称 | 主な機能 | 依存 | 推定規模 |
|------|------|---------|------|---------|
| 0 | 共通基盤 + 認証 | スキャフォールド, DB, Entity, 認証, Value Object | なし | 中 |
| 1 | 勤怠ドメイン | F-1〜F-8（打刻, 履歴, 集計, 修正） | Unit 0 | 大 |
| 2 | 社員管理 | F-9〜F-11（登録, 編集, 論理削除） | Unit 0 | 中 |

## 実装フロー

```
Phase 1:  Unit 0 を二人で実装（または一人がリード）
          ↓ 完了確認（テスト全パス + ログイン動作）
Phase 2:  Unit 1 と Unit 2 を別々に並行実装
          ↓ 各 Unit 完了
Phase 3:  結合テスト + 全体動作確認
```

## 並行実装の前提条件

Unit 1 と Unit 2 が衝突なく並行作業するために:

1. **Unit 0 完了後に分岐** — Entity / Repository / 認証が安定している
2. **パッケージを分離** — `attendance` パッケージ vs `employee` パッケージ
3. **共通コードの変更は相談** — SecurityFilterChain, GlobalExceptionHandler 等
4. **フロントエンドはページ単位で分離** — ルーティングが被らない

## コンフリクト回避ルール

| ファイル | 変更者 | 注意点 |
|---------|--------|--------|
| SecurityFilterChain | Unit 0 で確定 | URL パターン追加のみ許可 |
| GlobalExceptionHandler | Unit 0 で確定 | 新例外の追加のみ許可 |
| ナビゲーション | Unit 0 で骨格を作成 | メニュー項目の追加のみ |
| package.json / pom.xml | どちらか | 依存追加時は声掛け |
